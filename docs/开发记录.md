## 1. 项目启动与需求分析

### 1.1 项目背景

DreamGate 梦境档案馆是参加 **TRAE AI 创造力大赛** 的参赛作品，定位为「个人潜意识档案馆」Web App。核心理念是将碎片化的梦境记录转化为可累积、可检索、可解析的视觉化潜意识档案。

### 1.2 关键决策记录

| 决策项 | 决策结果 | 决策时间 | 决策依据 |
|--------|----------|----------|----------|
| 赛道选择 | 生活娱乐 | Spec v1 | 区别于效率工具赛道，梦境主题更契合生活娱乐，且视觉表现力强 |
| AI 图像生成方案 | Pollinations.ai flux（前端直连） | Spec v2 | 零成本、无需 API Key、URL 式直连、国内可访问 |
| AI 文本解析方案 | SiliconFlow Qwen2.5-7B（边缘函数代理） | Spec v3 | Groq 国内不可用，SiliconFlow 国内可达、免费额度充足 |
| 部署方案 | 腾讯云 EdgeOne Pages（主）+ Vercel（备） | Spec v3 | 国内 CDN 加速优先，Vercel 作为国际访问兜底 |
| 数据存储 | IndexedDB（idb 库） | Spec v1 | 纯前端、隐私友好、无需后端、容量充足 |
| 状态管理 | Zustand | Spec v1 | 轻量、TypeScript 友好、与 IndexedDB 同步简单 |
| 3D 渲染 | @react-three/fiber + drei + three | Spec v1 | 声明式 3D、React 生态、drei 提供 MeshReflectorMaterial 等高级材质 |
| 视觉风格 | noomo storytelling 灵感 | Spec v1 | 深色梦境感（#07070d）、雾气+粒子+颗粒、Cormorant Garamond + Noto Serif SC |
| 降级策略 | 4 个降级开关 + 第 12 天自检 | Spec v3 | 保 P0 闭环优先，3D 可降 2.5D、雾效着色器可降 CSS 渐变 |

### 1.3 需求分析核心洞察

- **痛点**：梦境记录碎片化、易遗忘、缺乏长期视角的模式识别
- **目标用户**：有自我探索意愿的普通人，非专业心理分析
- **核心价值**：AI 统一美学梦境图像 + 情绪/符号解析 + 3D 沉浸画廊 + 纵向模式识别 + 去社交化共享梦池
- **四重壁垒**：累积性（数据越用越有价值）、一致性（AI 统一美学）、社区性（梦池）、沉浸性（3D 画廊）

### 3.3 关键架构决策

1. **纯逻辑层与 UI 层分离**：`src/lib/*` 是纯 TypeScript 逻辑（无 React 依赖），`src/components/*` 和 `src/pages/*` 是 UI 层。Sub-Agent 实现时严禁修改 lib 层，确保逻辑稳定性。

2. **Zustand + IndexedDB 同步**：每个 store action（addDream/updateDream/deleteDream/setMeta）内部自动写 IndexedDB，UI 只操作 store，不直接调 db。

3. **降级开关双层设计**：
   - 编译期：`import.meta.env.VITE_DEGRADE_*`（构建时确定）
   - 运行时：`triggerDegradation(key)` 不可逆触发（第 12 天自检用）
   - 4 个开关：mobile2_5D / shareCard3D / fogShader / desktop3D

4. **AI 接入三层兜底**：
   - 第一层：Pollinations 图像 + SiliconFlow 解析（在线 + 同意）
   - 第二层：seedLibrary 占位图 + ruleParser 规则解析（离线或 AI 失败）
   - 第三层：DEFAULT_EMOTION（出神/0.4/neutral）+ 空符号列表（全失败）

5. **Sub-Agent 文件范围隔离**：并行 Task 时，每个 Sub-Agent 只能修改自己范围的文件，严禁触碰 lib/store/api 和其他 Task 的组件目录，避免并行冲突。

## 4. 开发过程中的重大问题及解决方案

### 4.1 PowerShell `&&` 语法错误

- **问题**：Sub-Agent 运行 `cd "d:/..." && npx tsc --noEmit` 时，PowerShell 报错 "The token '&&' is not a valid statement separator"。
- **原因**：PowerShell 不支持 `&&`/`||` 作为命令分隔符（这是 bash/cmd 语法）。
- **解决方案**：改用 Shell 工具的 `cwd` 参数指定工作目录，命令本身不带 `cd`。
- **影响范围**：多个 Sub-Agent 遇到此问题，已统一修复。

### 4.2 npm install ECONNRESET

- **问题**：首次 `npm install` 失败，网络错误 ECONNRESET（默认 npm registry 在国内不稳定）。
- **解决方案**：使用淘宝镜像 `npm install --registry=https://registry.npmmirror.com`，成功。
- **记录**：后续 Sub-Agent 不再需要 npm install（依赖已安装）。

### 4.3 Groq 国内不可用

- **问题**：Spec v2 原计划用 Groq 作为 AI 解析后端，但 Groq 在中国境内无法访问。
- **解决方案**：Spec v3 改用 SiliconFlow（硅基流动）Qwen2.5-7B-Instruct，国内可用、免费额度充足。
- **影响**：边缘函数 `api/llm.ts` 改为调用 SiliconFlow API。

### 4.4 Sub-Agent 并行文件冲突风险

- **问题**：Task 5/6/7/8 并行时，如果多个 Sub-Agent 修改同一文件（如 App.tsx），会导致冲突。
- **解决方案**：在 Sub-Agent prompt 中严格划定文件范围，每个 Task 只能修改自己的组件目录 + 指定页面。共用文件（lib/store/api）严禁修改。
- **结果**：4 个 Task 并行完成，零冲突。

### 4.5 Task 8 运行耗时过长 —— 第一性原理分析

- **现象**：Task 8（隐私与 AI 同意）的 Sub-Agent 运行时间明显长于并行批次（Task 5/6/7），tool_uses ~95，最终状态 `killed`。
- **结论先行**：长 runtime 是 Task 8「改共享文件」workload profile 的**固有属性**，不是低效，也不是失败。`killed` 是运行预算上限的副产品——代码已全部写完、tsc 通过后才被停止。

#### （a）Task 8 做什么

隐私与 AI 同意模块，覆盖三件事：全站免责页脚、隐私说明 modal（本地存储 / AI 外发范围 / 用户权利）、JSON 导出与清空二次确认。实际产出：

| 文件 | 行数 | 性质 | 复杂度 |
|------|------|------|--------|
| `src/components/Privacy/DisclaimerFooter.tsx` | 64 | 新建 | 低（fixed footer + localStorage） |
| `src/components/Privacy/PrivacyInfo.tsx` | 188 | 新建 | **高**（嵌套 AnimatePresence + 条件二次确认弹层 + 3 个 store hooks + 3 区块） |
| `src/components/Privacy/ExportButton.tsx` | 53 | 新建 | 低 |
| `src/components/Privacy/exportUtils.ts` | 47 | 新建 | 低 |
| `src/components/Privacy/index.ts` | — | 新建 | barrel |
| `src/App.tsx` | — | **修改** | 挂载 DisclaimerFooter（共享路由根） |
| `src/pages/DreamRoomPage.tsx` | — | **修改** | 挂载单梦 ExportButton |

**关键特征**：5 个新建 + **2 个改现有文件**（其中 `App.tsx` 是共享路由根，一处笔误全站崩）。

#### （b）第一性原理：Sub-Agent 运行时成本模型

把 Sub-Agent 的 runtime 拆到不可再分的最小单位：

> **runtime = Σ ( read_cost + modify_cost + create_cost + verify_cost + per_call_overhead )**

每个工具调用都贡献一项。Task 8 之所以最慢，逐项归因：

1. **「改现有文件」惩罚（主因）**：改 `App.tsx` 必须先 Read 全文找精确插入点 → Edit 受 `old_string` 唯一性约束（锚点不唯一要重试）→ 改完须重新 Read 验证 JSX 结构未坏。单次「改」≈ 3 次工具调用。对照 Task 5/6/7 多为「新建隔离目录文件」≈ 1 次 Write。**改共享文件的单位成本约为新建文件的 3×**。

2. **单文件复杂度最高**：`PrivacyInfo.tsx` 188 行（与 GenerationOrchestrator 并列最长），结构最复杂——嵌套 AnimatePresence + 条件二次确认弹层 + 3 个 store hooks + 3 区块。生成 + 自校验迭代次数最高，verify_cost 项最大。

3. **共享文件风险溢价**：`App.tsx` 一处笔误全站崩 → Sub-Agent 更谨慎，多轮 re-read 再下笔，进一步抬升 read_cost。

4. **tool_uses ~95 的构成**：主要由 read-modify-verify 循环占据，**非浪费**，是 workload profile 决定的下限。新建类任务（Task 5/6/7）tool_uses 远低于此，正因省去了「改」的 read-verify 往返。

#### （c）「killed」判定

Sub-Agent 在**代码已全部写完 + tsc EXIT_CODE=0 之后**、返回最终报告之前被停止。这是运行预算（tool_uses / 时长上限）触顶的副产品，**不是代码不完整**。手动复核确认：5 个新文件齐全、2 处现有文件改动正确、tsc 零错误。

#### （d）可复用教训

- 改共享文件的任务，单位 tool-call 成本 ≈ 3× 新建任务；
- 缓解三选一：(a) 在 Sub-Agent prompt 里预置精确 edit 锚点片段（省去 Read + 唯一性试错）；(b) 给改共享文件任务更高预算；(c) 共享文件（App.tsx / store / lib）改动收归主 Agent，Sub-Agent 只做隔离目录新建；
- `killed` ≠ 失败：先验证产物（tsc + 文件清单）再判定状态。

### 4.6 Pollinations 图像 CORS 问题

- **问题**：html-to-image 导出分享卡时，Pollinations 跨域图像可能导致 CORS 错误。
- **解决方案**：分享链接只编码 seedLibrary 图像路径（同源），不编码 Pollinations URL。导出时用 `cacheBust: true`，CORS 失败则提示用户截图。

### 4.7 降级开关命名不一致

- **问题**：Task 6 Sub-Agent 发现 spec 描述中降级开关名 `VITE_DEGRADE_3D_SHARE_CARD` 与 degradation.ts 实际定义 `VITE_DEGRADE_SHARE_CARD_3D` 不一致。
- **解决方案**：以 degradation.ts 实际定义为准（`VITE_DEGRADE_SHARE_CARD_3D`），Task 6 已正确使用。
- **记录**：spec 文档与代码不一致时，以代码为准。

### 4.8 Sub-Agent 编排经济学（结构性经验沉淀）

由 §4.5 Task 8 分析抽象出的可复用编排知识：

- **workload profile 决定 tool-call 成本**：
  - 「新建隔离目录文件」类任务（Task 5/6/7/9）≈ 快，单位成本 ≈ 1 次 Write；
  - 「改现有/共享文件」类任务（Task 8 改 App.tsx + DreamRoomPage.tsx）≈ 2-3× tool-call，因 Read → Edit（唯一性约束）→ re-Read 验证 往返。
- **编排启示**：
  1. 能并行的任务尽量隔离到独立目录，避免改共享文件；
  2. 共享文件（`App.tsx` / `store/` / `lib/` / `api/`）的改动收归主 Agent，Sub-Agent 只做隔离目录新建；
  3. 若必须由 Sub-Agent 改共享文件，在 prompt 里预置精确 edit 锚点片段（含足够上下文确保 `old_string` 唯一），省去 Read + 唯一性试错；
  4. 给改共享文件类任务预留更高 tool_uses 预算，或拆成「Sub-Agent 新建 + 主 Agent 接线」两段。
- **状态判定**：`killed` ≠ 失败。先验证产物（tsc + 文件清单 + 行数）再判定，避免误判返工。
- **工具使用教训（本次文档编辑）**：对同一文件并发发起多个 Edit 会触发竞态——后发的 Edit 基于旧快照覆盖，导致先发的整段新增丢失。同一文件的多次编辑必须串行。
