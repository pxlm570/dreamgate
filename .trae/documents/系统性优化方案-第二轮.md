# 梦阈 DreamGate 系统性优化方案（第二轮）

## Summary（概要）

上一轮 P0/P1（ErrorBoundary 兜底、生成/保存失败反馈、EdgeOne any 收窄、chunkSizeWarningLimit、degradation 单测）已全部落地，质量门全绿，lint 基线从 11 清零。

本轮基于 Phase 1 对**性能 / 资源 / UX / 可维护性**四维度的深入探查，聚焦 6 个**高价值、低风险、不改既有视觉与穿门逻辑**的优化项，分 P0/P1 两层。核心目标：

1. **修复真实 UX bug**：DreamRoomPage 直接访问（分享链接 / 刷新）时 `loadDreams` 未完成会误显「梦境不存在」，用户可能误判数据丢失而离开。
2. **修复 GPU 内存泄漏**：`useOptionalTexture` 与 `CorridorWalls` 的纹理在 deps 变化 / 卸载时未 dispose，反复切换 3D↔2.5D 会累积泄漏。
3. **性能与可维护性**：`ParallaxGallery` 无谓重算、`useDreamStore` 核心逻辑无测试覆盖。

严格遵守 HANDOFF 九条架构红线：不改门框视觉、不碰已否决方向、不动真连续穿门逻辑；所有改动后过质量门（check / test / lint / build）+ 真开浏览器实测。

## Current State Analysis（现状分析）

### UX（确认的真实 bug）
- **DreamRoomPage.tsx** L27 `dream = useDreamStore((s) => s.dreams.find((d) => d.id === id))`，**未读 `loaded`**。L48 `if (!dream)` 直接显示「梦境不存在」。
  - 触发场景：直接打开 `/#/dream/xxx`（分享链接 / 刷新 / 深链）。App.tsx L33-37 异步 `loadDreams()`，store 初始 `dreams: []`、`loaded: false`。DreamRoomPage 首次渲染时 dream=undefined → **闪现「梦境不存在」错误态**，待 IndexedDB 加载完成（通常 <100ms，慢设备更久）才纠正。
  - 危害：用户看到「梦境不存在 / 这场梦可能已被遗忘」可能直接离开，以为数据丢失。**这是影响分享链接可信度的真实 bug**。
- **ReportPage.tsx** L45-52：读 `dreams` 未读 `loaded`，`hasDreams = dreams.length > 0`。未加载时闪现空状态「你的情绪图谱尚未显现」。
- **DreamPoolPage.tsx** L32-50：读 `dreams` 未读 `loaded`。未加载时 `nextToShare=undefined` + `dreams.length===0` → 闪现「记录第一个梦」引导。池数据（POOL_DREAMS 静态）仍渲染，危害较轻。

### 资源（确认的 GPU 内存泄漏）
- **useOptionalTexture.ts** L14-37：`useEffect` cleanup 仅 `active = false`。
  - deps 变化（url/repeatX/repeatY 变）：旧纹理已在 state 中，新 effect 加载新纹理后 `setTex(newT)` 替换旧引用，**旧纹理 GPU 内存永不释放**（JS GC 回收对象但 WebGL 纹理内存泄漏）。
  - 卸载：已加载纹理在 state 中，cleanup 只置 `active=false`，**未 dispose**。
  - L27 `else t.dispose()` 只覆盖「卸载后加载才完成」的竞态，不覆盖「已加载后卸载/deps 变化」。
  - 调用方：`CorridorWorld` 用它加载 4 张走廊纹理（mist/wallPanel/floor/ceil）。URL 通常稳定，主要在卸载 / 降级切换时泄漏。
- **CorridorScene.tsx CorridorWalls** L143-189：6 个 `useMemo` 创建/克隆的纹理（endText / wallGrad / wallTex.left / wallTex.right / floorTex / ceilTex）。
  - R3F **不自动 dispose** 传给 material 的纹理（只 dispose material 本身与 geometry）。
  - `CorridorWalls` 卸载时（3D→2.5D 降级 / Canvas 卸载）这 6 个纹理全部泄漏。`wallPanel.clone()` 等克隆纹理有独立 GPU 上传，非共享。
  - 持久 Canvas 设计下卸载罕见，但降级触发时泄漏一次 ≈6 张纹理。低频但真实。

### 性能
- **ParallaxGallery.tsx** L23 `const recent = dreams.slice(-20).reverse()`：无 `useMemo`，每次渲染（含父组件任何 state 变化）都重算。虽 dreams 变化不频繁，但属无谓重算，修复零风险。
- **useSeedBootstrap** hooks.ts L68-82 第二个 effect 依赖 `[loaded, dreams, updateDream]`：每次 dreams 引用变化都 O(n) 扫描全部梦境检查种子迁移。有自终止逻辑（迁移后 URL 不再以 `/seeds/` 开头），但无 `migratingRef` 守卫。**评估后决定不改**：dreams 变化由用户操作触发（频率低），O(n) 在 n≤几十时开销可忽略，改动触及种子迁移逻辑风险 > 收益 → **不在本轮范围**。

### 可维护性
- **现有测试**（8 个）：degradation / edge-img-sync / edge-llm-sync / pool / shareUtils / patterns / seedLibrary / ruleParser。
- **未覆盖核心模块**：`ai.ts`（fetch 依赖，mock 成本高）、`db.ts`（IndexedDB 依赖）、`useDreamStore.ts`（依赖 db）。
- **useDreamStore.ts** 是全局状态中枢（dreams/meta/loaded/storageUnavailable + 5 个 action），其 `loadDreams` 失败降级、`addDream`/`updateDream`/`deleteDream` 状态转移无测试。可用 `vi.mock('@/lib/db')` + `vi.resetModules` 隔离测试，价值/成本比最高。

### P0-2：useOptionalTexture 纹理 dispose 修复（资源泄漏）

**文件**：`src/hooks/useOptionalTexture.ts`（修改）

**Why**：deps 变化 / 卸载时已加载纹理未 dispose → GPU 内存泄漏。

**What / How**：
- 新增 `loadedRef` 跟踪当前已加载纹理（供卸载时 dispose）。
- 加载成功回调：先存 `loadedRef.current = t`，再 `setTex(t)`；若 `active=false` 则 `t.dispose()`（既有逻辑）。
- deps 变化时新纹理加载完成后，dispose 上一张（`prev`）。
- 新增卸载专用 effect（`[]` 依赖）：cleanup 中 `loadedRef.current?.dispose()`。
- 完整实现：
  ```ts
  export function useOptionalTexture(
    url: string,
    repeatX = 1,
    repeatY = 1,
  ): THREE.Texture | null {
    const [tex, setTex] = useState<THREE.Texture | null>(null);
    const loadedRef = useRef<THREE.Texture | null>(null);

    useEffect(() => {
      let active = true;
      const loader = new THREE.TextureLoader();
      loader.load(
        url,
        (t) => {
          t.colorSpace = THREE.SRGBColorSpace;
          t.wrapS = THREE.MirroredRepeatWrapping;
          t.wrapT = THREE.MirroredRepeatWrapping;
          t.repeat.set(repeatX, repeatY);
          t.anisotropy = 8;
          if (active) {
            // deps 变化时释放上一张已不再使用的纹理 GPU 内存
            const prev = loadedRef.current;
            loadedRef.current = t;
            setTex(t);
            if (prev && prev !== t) prev.dispose();
          } else {
            t.dispose();
          }
        },
        undefined,
        () => { if (active) setTex(null); },
      );
      return () => { active = false; };
    }, [url, repeatX, repeatY]);

    // 卸载时释放当前纹理 GPU 内存（R3F 不自动 dispose 传给 material 的纹理）
    useEffect(() => {
      return () => {
        loadedRef.current?.dispose();
        loadedRef.current = null;
      };
    }, []);

    return tex;
  }
  ```
- **正确性论证**：
  - deps 变化：旧 effect cleanup 置 `active=false`（不 dispose，因纹理仍被 state 引用渲染）。新 effect 加载完成后 `setTex(newT)` 替换 state，此时 `prev`（旧纹理）不再被渲染 → `prev.dispose()` 安全，无「渲染已 dispose 纹理」的空窗。
  - 卸载：`[url,...]` effect cleanup 置 `active=false`（若加载中，回调到时 `t.dispose()`）；`[]` effect cleanup `loadedRef.current?.dispose()`（已加载的）。两者顺序：React 先跑 `[url,...]` cleanup 再跑 `[]` cleanup，无冲突。
  - 需 import `useRef`（当前仅 import 了 `useEffect, useState`）。

### P1-3：CorridorWalls 纹理 dispose（资源，纯增量）

**文件**：`src/components/Gallery/CorridorScene.tsx`（修改）

**Why**：`CorridorWalls` 内 6 个 `useMemo` 纹理（endText / wallGrad / wallTex.left / wallTex.right / floorTex / ceilTex）在卸载时未被 dispose（R3F 不自动 dispose 传给 material 的纹理）。3D→2.5D 降级或 Canvas 卸载时泄漏。

**What / How**：
- 在 `CorridorWalls` 组件内新增一个卸载专用 effect，用 ref 跟踪所有 useMemo 纹理并在卸载时 dispose：
  ```ts
  // 组件顶部
  const disposeRef = useRef<Array<() => void>>([]);
  // 每个 useMemo 末尾把 dispose 推入：
  //   const endText = useMemo(() => { ...; disposeRef.current.push(() => tex.dispose()); return ... }, []);
  // 实际更简洁：直接收集纹理引用
  ```
  具体实现：在 `CorridorWalls` 内用 `useRef<THREE.Texture[]>([])` 收集每次 useMemo 产生的纹理（含克隆的），卸载 effect 遍历 dispose。
  ```ts
  const texturesRef = useRef<THREE.Texture[]>([]);
  // endText useMemo:
  const endText = useMemo(() => {
    const r = makeEndTextTexture();
    texturesRef.current.push(r.tex);
    return r;
  }, []);
  // wallGrad / wallTex(left,right) / floorTex / ceilTex 同理 push
  // 卸载 cleanup
  useEffect(() => {
    return () => {
      for (const t of texturesRef.current) t.dispose();
      texturesRef.current = [];
    };
  }, []);
  ```
- **红线（严格遵守）**：仅在每个 useMemo 内追加 `texturesRef.current.push(...)` 一行 + 新增卸载 effect。**不改变纹理创建逻辑、不改变 repeat/wrap/offset 设置、不改变 material 引用**。纯增量 dispose，零视觉/行为影响。
- **风险控制**：`texturesRef.current.push` 在 useMemo 内执行（render 阶段），push 是幂等收集，React StrictMode 双调用会 push 两次但 dispose 同一引用是幂等的（three.js Texture.dispose 重复调用安全）。卸载 effect 用 `[]` 依赖，仅卸载时跑一次。

**Why**：全局状态中枢（loadDreams 降级 / addDream / updateDream / deleteDream 状态转移）无测试覆盖，是改动的回归安全网。

## Assumptions & Decisions（假设与决策）

1. **不引入新依赖**：所有修复用现有 API（three.js dispose / React useMemo / vitest vi.mock）。
2. **不改 useSeedBootstrap 第二个 effect**：评估后 O(n) 扫描在低频操作下开销可忽略，改动触及种子迁移逻辑风险 > 收益 → 不在本轮。
3. **不补 ai.ts 测试**：analyzeDream/generateArtifact 强 fetch 依赖，mock 成本高、回报边际（其内部兜底已有 ruleParser/seedLibrary 测试覆盖）→ 不在本轮。
4. **CorridorWalls dispose 纯增量**：只 push + 卸载 dispose，不改纹理创建/配置逻辑，视觉零影响。
5. **加载态用极简月亮呼吸点**：复用项目既有 `animate-pulse` + dreamgate-ethereal 配色，不引入新组件。
6. **遵守九条架构红线**：Canvas props 不变、crossing/stage 状态机不动、门框视觉不改、真连续穿门逻辑不碰。CorridorWalls 改动仅追加 dispose 收集。
