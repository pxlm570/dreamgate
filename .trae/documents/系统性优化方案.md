# 梦阈 DreamGate 系统性优化方案

## Summary（概要）

基于上轮 Phase 1 对性能 / UX / 可维护性三维度的探查，本方案聚焦**高价值、低风险、不改既有视觉与穿门逻辑**的优化项，分 P0/P1 两层执行。核心目标：

1. **健壮性兜底**：补全全局与局部 ErrorBoundary，消除 3D Canvas 崩溃导致的白屏风险；为生成流程与保存流程补用户级错误反馈。
2. **可维护性提升**：安全收窄 lint 基线中的 `any`（EdgeOne 运行时类型、SpeechRecognition 事件类型），不改变运行时行为。
3. **构建噪音治理**：调整 `chunkSizeWarningLimit` 消除 vendor-three 的告警噪音。

严格遵守 HANDOFF 九条架构红线：不改门框视觉、不碰已否决方向、不动真连续穿门逻辑；所有改动后过质量门（check / test / lint / build）+ 真开浏览器实测。

## Current State Analysis（现状分析）

### 性能
- **vite.config.ts** L65-76：`manualChunks` 已按库拆 `vendor-three` / `vendor-react` / `vendor-scroll` / `vendor-motion`，但未设 `chunkSizeWarningLimit`，`vendor-three ≈ 799kB(gzip 208kB)` 触发默认 500kB 警告，构建日志噪音。
- **CorridorScene.tsx** L9 `import * as THREE from "three"`：实际仅用 `CanvasTexture / SRGBColorSpace / MirroredRepeatWrapping / ClampToEdgeWrapping / RepeatWrapping / Texture / Points / PointsMaterial / PerspectiveCamera / MathUtils / AdditiveBlending`。tree-shaking 已能摇掉未用模块（生产构建无 dead code），改具名导入纯属可读性，收益/风险比低 → **不改**。
- **useDreamStore.ts**：selector 用法正确（每个 `useDreamStore((s) => s.xxx)` 返回稳定引用）。`dreams` 数组订阅在 dreams 变化时触发重渲染是预期行为（新梦境需显示）→ **无明显问题，不改**。
- **路由级懒加载**：App.tsx L12-17 已对全部页面 `React.lazy + Suspense`，首屏 bundle 已最小化。

### UX / 健壮性
- **App.tsx**：**无全局 ErrorBoundary**。任意组件渲染期抛错（尤其是 3D Canvas 的 WebGL context lost / shader 编译失败 / 纹理解码异常）会导致整树卸载白屏。这是**最高优先级**风险。
- **WorldPage.tsx** L360-367：`<Canvas>` 直接挂载，无局部边界。3D 崩溃会冒泡到全局。
- **GenerationOrchestrator.tsx** L132 `run().catch((err) => console.error(...))`：生成流程异常仅 console.error，**用户无任何感知**。虽然 `generateArtifact` 内部已有图像→种子图、解析→规则的完整兜底（L169-251 of ai.ts），但 `updateDream`（IndexedDB 写入）失败仍会走到这里，用户会卡在加载态。
- **RecordForm.tsx** L64-88 `handleSave`：`try/finally` 无 `catch`。`addDream` 内部 `db.addDream` 失败（IndexedDB 配额满 / 隐身模式 / 权限）会 throw，用户点「记下这场梦」后按钮恢复但数据未保存，**无任何反馈**。

### 可维护性
- **lint 基线 `any` 分布**（grep 确认，共 5 处实际 `any`）：
  - `edge-functions/api/ping.ts:4` `context: any`
  - `edge-functions/api/ping.ts:9` `Record<string, any>`
  - `edge-functions/api/llm.ts:22` `declare const AI: any`
  - `edge-functions/api/llm.ts:123` `context: any`
  - `edge-functions/api/llm.ts:178` `resp: any`
- 这些 `any` 集中在 EdgeOne 运行时注入对象（`AI` binding、`context`），无官方类型。可用**最小接口**安全收窄（只声明实际用到的字段），不改变运行时行为。
- `edge-functions/` 不在 tsconfig include（llm.ts L21 注释），`npm run check`(tsc) 不检查它，但 `npm run lint`(eslint) 会——收窄 any 主要是为 lint 干净。
- **双边缘函数同步**：已用 `edge-llm-sync.test.ts` + `edge-img-sync.test.ts` 守卫，不抽共享文件（避免 EdgeOne 打包风险）→ **保持现状**。

## Proposed Changes（改动清单）

### P0-1：通用 ErrorBoundary 组件 + 全局兜底（高优先级）

**文件**：`src/components/ui/ErrorBoundary.tsx`（新建）、`src/App.tsx`（修改）

**Why**：3D Canvas 崩溃或任意渲染期异常当前会白屏，无恢复路径。

**What / How**：
- 新建 `ErrorBoundary`（class component，React 仍需 class 做 error boundary）。Props：`children`、`fallback?: ReactNode`、`onReset?: () => void`。state：`{ error: Error | null }`。
- `static getDerivedStateFromError(error)` 置 error；`componentDidCatch` 仅 `console.error('[DreamGate]', error)`（不外发）。
- 渲染：有 error 时渲染 `fallback`（默认 = 深色背景 + 「梦境暂时失焦」文案 + 「重新进入」按钮，按钮调 `onReset` 或 `window.location.reload()`）。
- 在 `src/components/ui/index.ts` 导出。
- `App.tsx`：在 `<Router>` 外或 `<Suspense>` 外包一层全局 `ErrorBoundary`，兜底任意未捕获错误，避免白屏。全局兜底用默认 fallback（含 reload）。

**注意**：全局 ErrorBoundary 只兜"渲染期"错误；事件回调里的异步错误（如 `handleSave` 的 addDream 失败）不会被它捕获——那部分由 P0-3 处理。

### P0-2：WorldPage Canvas 局部 ErrorBoundary（3D 崩溃降级）

**文件**：`src/pages/WorldPage.tsx`（修改）

**Why**：3D 崩溃时不应整页白屏，应降级到已有 CSS fallback（`CssMirrorFallback`）或简单提示，保留 header/nav。

**What / How**：
- 在 L360-367 的 `<Canvas>` 外包一层局部 `ErrorBoundary`。
- 局部 fallback：复用 `CssMirrorFallback`（已从 `@/components/Gate` 导入，WorldPage.tsx L26），或一个极简深色占位 + 「3D 场景无法渲染，可继续记录梦境」文案 + 跳转「记录新梦」按钮。
- 决策：gate 阶段崩溃 → `CssMirrorFallback`（镜门已有 CSS 降级，视觉连贯）；corridor 阶段崩溃 → 提示文案 + 「记录新梦」入口（画廊 3D 崩了不阻塞记录）。
- 实现要点：用 `stage` prop 决定 fallback 形态。ErrorBoundary 的 `onReset` 清 error 后用户可重试挂载 Canvas。

**红线**：不改变 Canvas 的 props、不改变 `canvasOn` 判断逻辑、不碰 crossing/stage 状态机——只在 `<Canvas>` 外加一层包裹。

### P0-3：GenerationOrchestrator 生成失败的用户级反馈

**文件**：`src/components/Generation/GenerationOrchestrator.tsx`（修改）

**Why**：L132 `run().catch(console.error)` 让用户在 `updateDream` 失败时卡在加载态，无反馈无重试。

**What / How**：
- 新增 state：`const [fatalError, setFatalError] = useState(false)`。
- L132 改为：
  ```tsx
  run().catch((err) => {
    console.error("[DreamGate] generateArtifact failed:", err);
    setFatalError(true);
  });
  ```
- 在返回的 JSX 末尾加：当 `fatalError` 且 `!showArtifactView` 时，渲染一个轻量错误提示（深色卡片 + 「生成未能完成，可能是存储不可用」+ 「重试」按钮）。重试按钮：`setFatalError(false)` + 触发 effect 重跑（用一个 `retryToken` state 递增压入 effect 依赖）。
- 不引入 toast 系统（项目无现成 toast，自建一个全局 toast 超出本次范围）。错误就地上卡片即可。

**注意**：`generateArtifact` 内部兜底已完备（图像/解析都有 fallback），这里的 fatalError 仅覆盖 `updateDream`（IndexedDB 写入）失败这一窄路径。

### P0-4：RecordForm 保存失败的兜底反馈

**文件**：`src/components/Record/RecordForm.tsx`（修改）

**Why**：L64-88 `handleSave` 的 `try/finally` 无 `catch`，`addDream` 失败时按钮恢复但数据丢失，用户无感知。

**What / How**：
- 新增 state：`const [saveError, setSaveError] = useState(false)`。
- `handleSave` 加 `catch`：
  ```tsx
  } catch (err) {
    console.error("[DreamGate] addDream failed:", err);
    setSaveError(true);
  } finally {
    setSaving(false);
  }
  ```
- 在保存按钮下方（L141-145 的 `!hasText` 提示附近）加：当 `saveError` 时显示「保存失败——可能是浏览器存储受限（隐身模式 / 配额已满）。可复制梦境文本备用。」+ 「重试」按钮 + 一个「复制文本」按钮（`navigator.clipboard.writeText(rawText)`）。
- 重试：`setSaveError(false)` 后重新调 `handleSave`。

**注意**：不改动 store 的 `addDream` 逻辑（store 的 `storageUnavailable` 标记由 `loadDreams` 设置，`addDream` 失败此处就地处理即可，避免引入 store 层改动）。

### P1-1：EdgeOne 运行时 `any` 安全收窄

**文件**：`edge-functions/api/llm.ts`（修改）、`edge-functions/api/ping.ts`（修改）

**Why**：lint 基线 5 处 `any` 集中在 EdgeOne 运行时注入对象，可用最小接口安全收窄，提升类型安全与可读性，不改变运行时行为。

**What / How**：
- 在 `edge-functions/api/llm.ts` 顶部新增最小类型声明：
  ```ts
  /** EdgeOne 注入的 AI binding（运行时全局，最小类型声明） */
  interface EdgeAIChatCompletionResponse {
    choices?: { message?: { content?: string } }[];
  }
  interface EdgeAIBinding {
    chatCompletions: (opts: {
      model: string;
      messages: { role: string; content: string }[];
      stream?: boolean;
      temperature?: number;
      max_tokens?: number;
    }) => Promise<EdgeAIChatCompletionResponse | string | Response>;
  }
  declare const AI: EdgeAIBinding;

  /** EdgeOne Pages 边缘函数 context（最小类型） */
  interface EdgeOneContext {
    request: Request;
    env?: Record<string, string>;
  }
  ```
- L123 `onRequest(context: any)` → `onRequest(context: EdgeOneContext)`。
- L178 `let resp: any` → `let resp: Awaited<ReturnType<EdgeAIBinding['chatCompletions']>>`，或更简单 `let resp: EdgeAIChatCompletionResponse | string | Response`。
- L201-203 的 `resp?.choices?.[0]?.message?.content` 访问需配合 narrowing（用 `typeof resp === 'string'` / `resp instanceof Response` 已有，剩余对象形态访问 `resp.choices?.[0]?.message?.content` 在新类型下合法）。
- `edge-functions/api/ping.ts`：
  - L4 `context: any` → `context: EdgeOneContext`（需在 ping.ts 内复述最小声明，或接受重复——ping.ts 独立，不抽共享）。
  - L9 `Record<string, any>` → `Record<string, unknown>`。
  - L13 `Object.keys(AI)`、L14-15 `AI.chatCompletions` / `AI.chat` 访问需配合 `typeof AI !== 'undefined'` narrowing；`AI.chat` 不在最小接口里，需用 `(AI as { chat?: { completions?: { create?: unknown } } }).chat` 或扩展接口。决策：ping.ts 是探测脚本，保留宽松——把 `AI` 断言为 `unknown` 再 narrow，或直接 `as Record<string, unknown>`。

**红线**：edge-functions 不进 tsconfig（`npm run check` 不查），但进 eslint（`npm run lint`）。改动后必须 `npm run lint` 确认基线 problems 数下降且无新增。运行时行为零改变（纯类型注解）。

## Assumptions & Decisions（假设与决策）

1. **不引入新依赖**：ErrorBoundary 自建（React class component），不装 `react-error-boundary`。无现成 toast 库，错误反馈用就地上卡片，不引入 toast 系统。
2. **edge-functions 类型收窄纯注解**：EdgeOne 运行时注入对象无官方类型，最小接口只声明实际用到的字段；不抽共享类型文件（避免 EdgeOne 打包风险，与现有双边缘函数同步策略一致）。
3. **不碰 selector / `* as THREE`**：评估后无明显收益，改动风险 > 价值。
4. **不进一步拆 manualChunks**：three / @react-three / postprocessing 同页加载，拆分缓存收益边际。
5. **全局 ErrorBoundary 只兜渲染期错误**：异步/事件回调错误由 P0-3/P0-4 就地处理。
6. **遵守九条架构红线**：Canvas props 不变、crossing/stage 状态机不动、门框视觉不改、真连续穿门逻辑不碰。
